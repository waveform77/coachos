# CoachOS — Руководство по защите диплома

## Как объяснить проект на защите

Этот документ содержит готовые объяснения архитектуры и функциональности, адаптированные для дипломной комиссии.

---

## 1. Зачем нужен проект (простыми словами)

**Вопрос комиссии**: «Какую проблему решает ваш проект?»

**Ответ**:
> Типичная футбольная академия сегодня работает в Excel, Word и Telegram-чатах. Тренер тратит 3–5 часов в неделю на административную работу вместо тренировочного процесса. Прогресс игрока нигде не фиксируется систематически. Родители не понимают, как развивается ребёнок.
>
> CoachOS — это цифровая платформа, которая переносит всю работу академии в единое пространство: планирование тренировок, оценка игроков, аналитика прогресса, связь с родителями. Это снижает административную нагрузку тренера в разы и позволяет объективно измерять развитие каждого игрока.

## 2. Почему Go и Fiber для backend?

**Вопрос**: «Почему вы выбрали именно Go, а не Python/Node.js?»

**Ответ**:
> Go — это компилируемый, статически типизированный язык с встроенной поддержкой конкурентности. Для REST API он обеспечивает:
>
> 1. **Производительность**: обрабатывает тысячи запросов в секунду на скромном железе — важно для академии с сотнями пользователей
> 2. **Надёжность**: строгая типизация исключает целый класс ошибок ещё на этапе компиляции
> 3. **Малый footprint**: бинарный файл ~15MB, контейнер ~20MB — дешевле в облаке
>
> Fiber — это высокопроизводительный web framework для Go, вдохновлённый Express.js. Он построен на fasthttp и показывает одни из лучших benchmark-результатов среди Go-фреймворков.

## 3. Почему PostgreSQL?

**Вопрос**: «Почему реляционная БД, а не NoSQL?»

**Ответ**:
> Данные футбольной академии имеют сложные связи: игрок ↔ команда ↔ тренировка ↔ посещаемость ↔ оценки. Реляционная модель идеально подходит для:
>
> 1. **Целостности данных**: Foreign keys и constraints гарантируют, что не может существовать оценка без игрока
> 2. **Сложных аналитических запросов**: расчёт статистики посещаемости, агрегация оценок — всё через SQL JOIN/GROUP BY
> 3. **ACID транзакций**: атомарность важна при отметке посещаемости сразу для 20 игроков
>
> PostgreSQL также поддерживает JSONB для гибких метаданных (settings, AI-ответы) без отдельных таблиц.

## 4. Почему React + TypeScript для frontend?

**Вопрос**: «Почему не Vue или Angular?»

**Ответ**:
> React — лидирующая UI-библиотека с огромной экосистемой. TypeScript добавляет строгую типизацию, что критично для проекта с 15+ доменными сущностями и 5 ролями пользователей.
>
> Выбор стека:
> - **Vite**: сборщик нового поколения, hot reload за <50ms
> - **TanStack Query**: управление server state — кэширование, фоновые обновления, состояния загрузки
> - **shadcn/ui + Tailwind**: готовые accessible компоненты + utility-first CSS = быстрая разработка красивого UI
> - **React Hook Form + Zod**: надёжная валидация форм с type safety

## 5. Что такое Clean Architecture?

**Вопрос**: «Объясните архитектурный подход backend»

**Ответ**:
> Clean Architecture — принцип организации кода, при котором зависимости направлены строго внутрь:
>
> ```
> HTTP (handler) → Business Logic (service) → Data Access (repository) → Database
> ```
>
> **Главное правило**: бизнес-логика не знает о HTTP и БД. Если мы сменим PostgreSQL на MongoDB — сервисы останутся нетронутыми. Если сменим Fiber на gRPC — бизнес-логика не изменится.
>
> **Практическая польза для диплома**:
> - Каждый слой легко тестировать изолированно
> - Код понятен: обработчик только парсит запрос, сервис только содержит бизнес-логику
> - Масштабируемость: легко выделить модуль в отдельный микросервис

## 6. Как работает авторизация?

**Вопрос**: «Как реализована система авторизации?»

**Ответ**:
> Используется **JWT (JSON Web Token)** в сочетании с **Refresh Token Rotation**.
>
> **Схема**:
> 1. Пользователь вводит email/password
> 2. Сервер проверяет пароль через bcrypt и выдаёт:
>    - **Access Token** (JWT, живёт 15 минут) — хранится в памяти браузера
>    - **Refresh Token** (случайный UUID, живёт 7 дней) — httpOnly cookie (недоступен JavaScript)
>
> 3. Каждый API-запрос несёт Access Token в заголовке Authorization
> 4. Когда Access Token истекает (401), браузер автоматически запрашивает новый через Refresh Token
>
> **Защита от кражи токена** (Token Theft Protection):
> Если злоумышленник украл Refresh Token и попытался его использовать повторно — сервер детектирует «reuse» и немедленно инвалидирует все токены этой «семьи» (family_id). Законный пользователь будет разлогинен и уведомлён.

## 7. Как устроены роли?

**Вопрос**: «Расскажите о системе ролей (RBAC)»

**Ответ**:
> RBAC — Role-Based Access Control. В системе 5 ролей с разными правами доступа:
>
> | Роль | Доступ |
> |---|---|
> | Admin | Полное управление клубом |
> | Coach | Тренировки, команды, оценки |
> | Player | Только свой профиль и расписание |
> | Parent | Профиль своего ребёнка |
> | Analyst | Только чтение аналитики |
>
> На backend каждый защищённый endpoint имеет middleware `RequireRole(roles...)`, который проверяет роль из JWT. Дополнительно проверяется принадлежность к клубу — тренер из академии A не может видеть данные академии B.
>
> На frontend компонент `<RoleGuard>` скрывает UI-элементы, недоступные текущей роли.

## 8. Как считается Player Development Index?

**Вопрос**: «Что такое Player Development Index и как он работает?»

**Ответ**:
> PDI — нормированный индекс от 0 до 100, отражающий комплексный прогресс игрока.
>
> **Формула**:
> ```
> PDI = (Посещаемость% × 0.20) + (Средняя_оценка × 10 × 0.50) + (Цели_достигнуты% × 0.30)
> ```
>
> Весовые коэффициенты выбраны так, что:
> - Оценки тренера важнее всего (50%) — это экспертная оценка
> - Посещаемость (20%) — без тренировок нет прогресса
> - Достижение целей (30%) — мотивационная составляющая
>
> **Интерпретация**:
> - 0–39: Требует внимания (красный)
> - 40–69: Нормальное развитие (жёлтый)
> - 70–100: Отличный прогресс (зелёный)
>
> Это позволяет тренеру и родителям объективно оценить, насколько ребёнок прогрессирует.

## 9. Как работает AI-ассистент?

**Вопрос**: «Что такое AI Coach Assistant?»

**Ответ**:
> AI Assistant — это модуль, который помогает тренеру планировать тренировки и анализировать прогресс игроков.
>
> **Архитектура**: система построена через интерфейс `AIProvider`, что позволяет легко менять реализацию:
>
> ```
> interface AIProvider {
>     GenerateTrainingPlan(goal, duration, focus) → plan
>     RecommendExercises(weakSkill, player) → exercises
>     AnalyzePlayer(assessments, attendance) → strengths + weaknesses
>     SummarizeProgress(period) → human-readable summary
> }
> ```
>
> **MVP реализация**: MockProvider — правила-based логика без внешнего API. Например: если цель «улучшить прессинг», выдаётся план с упражнениями категории «tactics» и «physical». Если у игрока слабая техника (оценка < 5) — предлагаются упражнения из библиотеки категории «technique».
>
> **Подключение реального AI** (будущее): достаточно реализовать `OpenAIProvider` с теми же методами интерфейса. Вся система продолжит работать без изменений в других слоях.

## 10. Как масштабировать проект?

**Вопрос**: «Как система будет масштабироваться при росте нагрузки?»

**Ответ**:
> Проект построен по принципу **monolith-first, microservice-ready**:
>
> **Сейчас (MVP)**:
> - Единый Go monolith + PostgreSQL
> - Горизонтальное масштабирование monolith за load balancer
>
> **При росте (Phase 2)**:
> - Добавляем Redis для кэширования аналитических запросов
> - Read replicas PostgreSQL для аналитики
>
> **При значительном росте (Phase 3)**:
> - Модульная структура monolith позволяет вынести `analytics-service` как отдельный сервис (он самый тяжёлый)
> - Event-driven уведомления через message broker
>
> **Принципиально важно**: Clean Architecture гарантирует, что выделение в микросервис — это замена одного слоя repository, без изменения бизнес-логики.

## Демо-сценарий для защиты

### Порядок показа (10–15 минут)

1. **Landing page** (30 сек) — «Вот что видит потенциальный клиент»

2. **Логин как тренер** (1 мин) — «Роль coach, Coach Command Center»
   - Показать: today's sessions, absent today, players at risk

3. **Создание тренировки** (2 мин)
   - Открыть календарь
   - Создать сессию на сегодня
   - Добавить блоки (warmup + main + cooldown)
   - Добавить упражнения из библиотеки
   - Показать расчёт нагрузки

4. **Профиль игрока** (2 мин)
   - Открыть профиль любого игрока
   - Показать PDI gauge
   - Показать Radar Chart навыков
   - Добавить оценку → PDI пересчитывается

5. **AI Assistant** (2 мин)
   - «Сгенерируй тренировку на тему: улучшить прессинг»
   - Показать структурированный ответ

6. **Аналитика** (1 мин)
   - Графики посещаемости
   - Training load chart

7. **Смена роли — Parent** (1 мин)
   - «Вот что видит родитель»
   - Parent Transparency Mode: PDI, посещаемость, recommendations

8. **Архитектура** (2 мин)
   - Показать структуру папок в IDE
   - Открыть domain/models.go — «чистые модели без HTTP»
   - Открыть service/player_service.go — «бизнес-логика без SQL»
   - Открыть handler/player_handler.go — «только HTTP, нет бизнес-логики»

## Возможные вопросы комиссии и ответы

**Q: Почему не использовали GraphQL?**
> REST достаточен для MVP. GraphQL добавляет сложность без очевидной пользы при нашем паттерне запросов. Архитектура позволяет добавить GraphQL слой поверх тех же сервисов в будущем.

**Q: Как тестировали проект?**
> Написаны unit-тесты для критичной бизнес-логики (PDI расчёт, auth flow, RBAC), integration tests для репозиториев с реальной PostgreSQL через testcontainers, и E2E smoke тесты через Playwright.

**Q: Есть ли у проекта коммерческий потенциал?**
> Рынок спортивного ПО оценивается в $6 млрд и растёт. Конкуренты в России (TeamSnap, SportsEngine) не имеют русскоязычного UX и глубокой вертикали для футбольных академий. Модель — B2B SaaS: freemium до 20 игроков, платные планы от $49/месяц.

**Q: Что бы вы изменили, начав заново?**
> Начал бы с sqlc вместо GORM для аналитических запросов — более предсказуемые SQL и type safety. Но для MVP GORM оправдан: сэкономил несколько дней на репозиториях.
